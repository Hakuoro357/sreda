# Data Security And Isolation

## Главный принцип

Нельзя смешивать:
- agent state;
- продуктовые данные;
- персональные данные;
- системные секреты.

Даже если все физически лежит в одном `PostgreSQL`, логически это должны быть разные контуры.

## Базовые сущности изоляции

- `tenant`
- `workspace`
- `user`
- `assistant`
- `thread`

Все ключевые записи должны быть привязаны минимум к:
- `tenant_id`
- `workspace_id`

## Что хранить отдельно

### Product data

- пользователи;
- memberships;
- каналы;
- настройки;
- расписания;
- источники.

### Agent runtime data

- threads;
- runs;
- checkpoints;
- summaries;
- decisions.

### Sensitive data

- профили;
- контактные данные;
- документы;
- токены интеграций;
- любые PII-поля.

## Шифрование

Для MVP:
- шифровать чувствительные поля на уровне приложения;
- использовать `AES-256-GCM`;
- ключ хранить вне кода и конфигов.

Для развития:
- перейти к `per-tenant` ключам;
- для чувствительных клиентов поддержать `per-workspace` или `per-installation` ключ;
- для on-prem использовать KMS/Vault клиента.

## Практические правила

- не класть сырой PII в свободную память агента;
- не давать LLM полный профиль без policy check;
- получать чувствительные данные через отдельный tool;
- логировать все чтения и изменения чувствительных сущностей;
- любые risky actions проводить через approval.

## Изоляция в Postgres

Для SaaS нужно закладывать:
- `RLS`;
- обязательные tenant filters;
- уникальные индексы с учетом `tenant_id`;
- namespaces для runtime state;
- отдельные таблицы аудита.

## On-prem модель

Для особо чувствительных клиентов нужно предусмотреть:
- single-tenant deployment;
- отдельную БД;
- отдельные ключи;
- локальную модель или локальный model gateway;
- отсутствие смешения данных с SaaS-контуром.
