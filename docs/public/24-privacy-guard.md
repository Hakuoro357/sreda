# Privacy Guard для Sreda

Этот документ фиксирует минимальный `privacy guard` для `Sreda` на первом этапе.

## Решение

На MVP используем только:
- `regex`
- собственные правила
- без сторонних библиотек

## Зачем нужен privacy guard

Все входящие сообщения из `Telegram` потенциально могут содержать:
- телефоны
- email
- адреса
- номера лицевых счетов
- токены
- логины
- пароли
- другие чувствительные данные

Поэтому raw-текст пользователя не должен уходить в `LLM` напрямую.

## Базовый flow

1. Приходит raw message от пользователя.
2. Сообщение проходит через `privacy_guard`.
3. `privacy_guard` строит безопасную версию текста.
4. В `LLM` уходит только `sanitized_text`.
5. Исходный текст не отправляется в `LLM`.

## Что должен делать privacy guard

`privacy_guard` должен:
- находить чувствительные фрагменты в тексте;
- заменять их на безопасные маркеры;
- возвращать sanitized-версию текста;
- возвращать список найденных сущностей;
- выставлять флаг `contains_sensitive_data`.

## Минимальные типы чувствительных данных

На первом этапе закрываем:
- телефон -> `[phone]`
- email -> `[email]`
- URL с токеном или секретом -> `[url]`
- bearer/token/api key -> `[secret]`
- номер лицевого счета -> `[account_number]`
- длинные числовые идентификаторы -> `[number]`
- пароль -> `[password]`
- логин -> `[login]`
- код подтверждения -> `[verification_code]`

## Источники правил

### 1. Общие regex-правила

Используются для:
- телефонов;
- email;
- URL;
- токенов;
- длинных цифровых строк.

### 2. Доменные правила Sreda

Отдельно добавляются правила под наш контур:
- `номер лицевого счета`
- `логин EDS`
- `пароль EDS`
- `код подтверждения`
- текст после ключевых слов `логин`, `пароль`, `токен`

## Что уходит в LLM

В `LLM` уходит только sanitized-представление.

Пример:

Raw:

```text
мой новый телефон +7 999 123-45-67, запомни его
```

Sanitized:

```text
мой новый телефон [phone], запомни его
```

Если после этого `LLM` определит интент `memory_update`, само исходное значение телефона должно использоваться уже не моделью, а кодом после явного подтверждения пользователя.

## Что не должен делать LLM

`LLM` не должен видеть:
- пароли;
- токены;
- логины к внешним кабинетам;
- email;
- телефоны;
- номера лицевых счетов;
- документы и реквизиты;
- другие чувствительные данные, если они написаны в сообщении.

## Что хранить

На MVP допустим такой подход:
- raw message не отправляется в `LLM`;
- в `LLM` идет только `sanitized_text`;
- исходные значения используются только внутри runtime-кода, если это нужно для конкретного действия.

## Плюсы решения

- бесплатно;
- просто реализовать;
- предсказуемо;
- легко отлаживать;
- легко расширять новыми правилами.

## Минусы решения

- не поймает все неявные персональные данные;
- хуже работает на свободных формулировках;
- качество зависит от качества набора regex и доменных правил.

## Итог

Для MVP `Sreda` принимается такой выбор:
- отдельный модуль `privacy_guard.py`;
- только `regex` и свои правила;
- ставится перед любым вызовом `LLM`;
- raw user message в `LLM` не отправляется.

## Текущая реализация

На текущем этапе это уже реализовано в коде:
- ядро: модуль `privacy_guard` в core runtime
- текущая интеграция с `LLM` в `EDS monitor`: summary layer feature module

Сейчас через `privacy_guard` перед `LLM` уже проходят:
- `description`
- `last_answer`
- и остальные текстовые поля summary payload, которые уходят в модель

Это не закрывает еще весь будущий Telegram assistant flow, но задает общий механизм, который дальше должен использоваться и для входящих пользовательских сообщений.
