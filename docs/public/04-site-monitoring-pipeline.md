# EDS Monitoring Pipeline

## Задача

Нужен сценарий, где система по расписанию:
- заходит на сайт;
- находит новые данные;
- извлекает полезную информацию;
- делает summary;
- при необходимости дополняет;
- отправляет результат в Telegram.

## Важное решение

Для MVP это не нужно реализовывать как полностью автономного агента.

Достаточно модели:
- `scheduled job`
- `deterministic parsing`
- `LangGraph analysis step`
- `outbox delivery`

То есть агент работает как этап пайплайна, а не как самостоятельная сущность без границ.

## Что делает обычный код

- запускает задачу по расписанию;
- получает страницу или API-ответ;
- парсит HTML/JSON;
- извлекает кандидатов на новые записи;
- считает fingerprint;
- делает dedup;
- сохраняет сырой и нормализованный результат.

## Что делает LangGraph

- классифицирует запись;
- определяет важность;
- делает краткое резюме;
- при необходимости добавляет пояснение;
- решает, отправлять ли это в канал;
- формирует итоговое сообщение.

## Рекомендуемый flow

1. `prepare_item`
2. `classify_item`
3. `summarize_item`
4. `score_item`
5. `decide_delivery`
6. `format_message`

## Почему это лучше автономного агента на MVP

- проще контролировать;
- легче отлаживать;
- меньше вероятность runaway-поведения;
- легче обеспечить предсказуемость;
- проще объяснить клиенту, как именно система принимает решение;
- меньше рисков при работе с чувствительными данными.

## Антиспам-ограничения

Нужны сразу:
- deduplication по fingerprint;
- cooldown для повторяющихся событий;
- limit сообщений за окно времени;
- возможность агрегировать несколько событий в одну сводку;
- порог отправки по приоритету.

## Пример MVP scope

Первый релиз лучше ограничить:
- один источник;
- один тип контента;
- один Telegram-канал;
- один сценарий summary;
- без enrichment через внешние сервисы;
- без обратных действий на сайт.
