# Why LangGraph

## Что такое LangGraph

`LangGraph` это runtime для stateful и long-running агентных workflow.

Он дает базовые примитивы:
- `graph` из узлов и переходов;
- `state` для накопления состояния;
- `threads` для отдельных сессий;
- `checkpoints` для продолжения после паузы или сбоя;
- `interrupts` для human-in-the-loop;
- `store` для долгой памяти.

## Почему он подходит под Sreda

Для нашего кейса нужны не просто ответы модели, а управляемые процессы:
- пользовательские диалоги;
- workflow по расписанию;
- обработка событий;
- согласование рискованных действий;
- долговременная память;
- переносимость между SaaS и on-prem.

LangGraph подходит, потому что:
- хорошо решает stateful execution;
- позволяет останавливать и возобновлять выполнение;
- подходит для фоновых workflow;
- дает понятную архитектурную модель для сложных agent-процессов;
- не навязывает готовый продуктовый слой, поэтому можно собрать свою платформу.

## Что важно понимать

LangGraph не решает из коробки:
- multitenant auth;
- RBAC;
- tenant billing;
- хранение PII по правилам бизнеса;
- Telegram/MAX gateway;
- продуктовую админку.

Значит, его нужно использовать как ядро orchestration, а не как весь продукт.

## Почему не SuperAgentX как основа

`SuperAgentX` хорош как легкий конструктор, но для нашей задачи важнее:
- зрелое состояние runtime;
- надежная работа с long-running процессами;
- interruptions;
- persistence;
- более сильная база для production-сценариев.

Поэтому для ядра лучше `LangGraph`.

## Почему Python

Под наш кейс `Python` подходит лучше, потому что:
- сильнее AI-экосистема;
- больше примеров и библиотек;
- проще разрабатывать с помощью ИИ;
- разница в чистом runtime обычно не становится ключевой, потому что узкое место чаще в LLM, БД и внешних API.
