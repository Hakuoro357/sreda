# Decisions

## Цель

Зафиксировать стартовые архитектурные решения в формате коротких ADR, чтобы команда не спорила заново о базовых вещах на каждом этапе.

## ADR-001. Ядро агентного runtime

### Решение

Используем `LangGraph` как основной runtime для agent workflow.

### Почему

- нужен stateful execution;
- нужны long-running flow;
- нужен `interrupt` для approval;
- нужен устойчивый фундамент под SaaS и on-prem.

### Последствия

- orchestration будет строиться вокруг graph/state/checkpoints;
- platform layer придется делать отдельно;
- часть команды должна освоить подход LangGraph.

## ADR-002. Основной язык

### Решение

Основной язык backend и agent runtime: `Python`.

### Почему

- сильнее AI-экосистема;
- удобнее работать с LangGraph;
- больше примеров;
- проще разрабатывать с помощью ИИ.

### Последствия

- web-компоненты при необходимости можно вынести отдельно;
- интеграции и runtime будут сосредоточены в Python-кодовой базе.

## ADR-003. Канал первого запуска

### Решение

На MVP используем только `Telegram`.

### Почему

- уменьшает сложность интеграций;
- позволяет быстрее собрать pilot;
- снижает объем edge-case логики.

### Последствия

- `MAX` переносится на следующий этап;
- модель каналов должна изначально оставаться расширяемой.

## ADR-004. Стек хранения на MVP

### Решение

На MVP основное хранилище: только `PostgreSQL`, без `Redis`.

### Почему

- проще инфраструктура;
- проще on-prem;
- достаточно для jobs, outbox, runtime state и audit на раннем этапе.

### Последствия

- jobs queue строится на `FOR UPDATE SKIP LOCKED`;
- rate limiting и ephemeral counters будут упрощенными;
- при росте нагрузки Redis можно добавить позже.

## ADR-005. Формат фоновой автоматизации

### Решение

На MVP не делаем широких автономных фоновых агентов. Делаем `scheduled workflows` и `background jobs` с агентным шагом.

### Почему

- проще отладка;
- выше предсказуемость;
- ниже риск runaway-поведения;
- легче обеспечить безопасность.

### Последствия

- cron/scheduler и deterministic parsing остаются вне LLM;
- LangGraph используется в аналитических шагах pipeline.

## ADR-006. Модель изоляции данных

### Решение

Во всех ключевых сущностях используем `tenant_id` и `workspace_id`.

### Почему

- это основа мультиарендности;
- без этого трудно перейти к командным сценариям и on-prem профилям.

### Последствия

- даже если workspace сначала один, поле все равно сохраняем;
- в БД закладывается `RLS`;
- фильтры по tenant/workspace обязательны во всех сервисах.

## ADR-007. Хранение чувствительных данных

### Решение

Чувствительные данные храним отдельно от runtime state и шифруем на уровне приложения.

### Почему

- нельзя смешивать PII и память агента;
- это снижает риск утечки в prompt/state;
- так проще обосновывать безопасность клиентам.

### Последствия

- runtime использует только безопасные summary;
- для чтения sensitive данных нужен отдельный access path;
- в будущем можно перейти на per-tenant keys.

## ADR-008. Доставка сообщений

### Решение

Все исходящие сообщения отправляются через `outbox pattern`.

### Почему

- это повышает надежность;
- упрощает retries;
- помогает расследовать инциденты.

### Последствия

- worker не должен отправлять внешние сообщения напрямую из середины flow;
- доставка становится отдельной операционной зоной.

## ADR-009. Хранение данных EDS monitor

### Решение

Полные данные заявок на стороне `Sreda` не хранятся.

### Почему

- минимизируем хранение чувствительных данных;
- уменьшаем поверхность утечки;
- сохраняем только признаки изменения и технический state.

### Последствия

- полная карточка заявки используется только transiently во время обработки;
- в БД хранятся fingerprint, event metadata, delivery metadata и технический state;
- graph и runtime не должны складывать полные payload заявок в постоянные checkpoints.

## ADR-010. Язык integration layer для EDS monitor

### Решение

Текущий monitoring integration layer сразу переписывается на `Python`.

### Почему

- нужен единый стек;
- проще сопровождение;
- проще связать с новым backend и LangGraph runtime.

### Последствия

- интеграции и runtime остаются в одном языке;
- на старте больше инфраструктурной работы, но чище итоговая архитектура.

## ADR-011. Capability model

### Решение

Платные и дополнительные функции системы управляются через capability flags на уровне `tenant`.

### Почему

- позже фич будет больше одной;
- не нужно зашивать billing в runtime;
- удобно включать и выключать функции вручную на MVP.

### Последствия

- появляется таблица `tenant_features`;
- API, scheduler и worker должны проверять feature availability;
- `eds_monitor` становится первой feature-gated capability.

## ADR-012. Графы MVP

### Решение

На MVP делаем два основных графа:
- assistant flow;
- `eds_monitor` flow.

### Почему

- этого достаточно для первого полезного релиза;
- не нужен multi-agent zoo;
- проще тестировать и объяснять.

### Последствия

- supervisor/subgraphs откладываются;
- архитектура остается открытой к расширению, но не перегружена.

## ADR-013. Формат поставки первого enterprise-варианта

### Решение

On-prem рассматривается как `single-tenant deployment profile`, а не как отдельный продуктовый форк.

### Почему

- это снижает расхождение кодовой базы;
- проще сопровождать;
- легче переносить улучшения из SaaS в коробку.

### Последствия

- конфигурация должна быть profile-driven;
- секреты, ключи и storage должны подменяться через deployment profile.
