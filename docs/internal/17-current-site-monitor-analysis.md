# Current EDS Monitor Analysis

## Цель

Зафиксировать текущие инварианты `eds_monitor`, которые уже считаются обязательными для `Среды`.

## Что уже сделано правильно

### 1. Главная единица изоляции

Текущая система правильно выбрала единицу изоляции: `accountId`.

Именно по `accountId` уже разделены:
- логин и пароль;
- session storage;
- poll state;
- delivery state;
- runtime logs;
- список уже известных заявок.

Это очень важный инвариант. При переносе в `LangGraph` его нельзя терять.

## 2. Вся опасная и детерминированная логика уже вынесена из LLM

Текущая система не делает типичную ошибку "агент пусть сам всё решает".

Сейчас:
- логин в сайт делает код;
- роль оператора активирует код;
- список заявок получает код;
- детали заявки получает код;
- state diff считает код;
- формат нормализованной заявки собирает код;
- Telegram delivery делает код.

LLM используется только для анализа одной уже нормализованной заявки.

Это правильная граница и в `LangGraph` её нужно сохранить.

## 3. Уже есть хорошие operational инварианты

### Session reuse

Текущий runtime:
- используется Playwright context;
- storage state хранится отдельно по `accountId`;
- если cookies валидны, повторный login не нужен;
- после логина обязательно вызывается `POST /api/user-role?role=ROLE_EDC_OPERATOR`.

### Poll state

Состояние заявок:
- fingerprint строится по значимым полям;
- есть разделение на `newClaims`, `changedClaims`, `unchangedClaims`;
- состояние сохраняет `firstSeenAt`, `lastSeenAt`, статус и hash.

### Delivery state

Доставка:
- отдельно хранится delivery record по `claimId + chatId`;
- повторная отправка заменяется на `editMessageText`, если текст уже был отправлен;
- фото не дублируются, если уже были отправлены.

Это не просто "уведомлялка", а уже почти полноценный idempotent delivery контур.

## Текущий поток выполнения

Текущий основной сценарий такой:

1. загрузить конфиг и секреты;
2. выбрать accounts;
3. по каждому `accountId`:
- прочитать state;
- открыть или восстановить session;
- активировать operator role;
- забрать список заявок;
- классифицировать `new/changed/unchanged`;
- для новых заявок получить detail;
- нормализовать detail;
- отдать нормализованную заявку в LLM-анализ;
- собрать текст уведомления;
- при необходимости скачать фото;
- отправить или обновить Telegram-сообщение;
- обновить delivery state;
- обновить poll state.

## Слои текущей реализации

### Config layer

- читает secrets и runtime settings;
- валидирует accounts;
- нормализует пути и таймауты.

### Integration layer

- логинится в EDS;
- восстанавливает session;
- активирует роль оператора;
- вызывает list/detail/works API;
- скачивает фото и вложения.

### State layer

- строит fingerprint;
- классифицирует `new/changed/unchanged`;
- хранит технический state;
- хранит delivery state.

### Analysis layer

- готовит compact payload для модели;
- вызывает LLM;
- валидирует structured result;
- возвращает summary и delivery decision.

### Delivery layer

- формирует Telegram HTML text;
- докладывает фото;
- обеспечивает idempotent delivery semantics.

## Что в текущей реализации временное

### File-based state

Сейчас часть состояния еще может жить в файлах:
- session state;
- временные delivery артефакты;
- runtime helper state.

Для платформы это надо приводить к БД и явным runtime directories.

### Changed claims

`changedClaims` уже считаются отдельным сценарием.

Это важно: `eds_monitor` изначально должен поддерживать не только новые заявки, но и повторные уведомления при значимом изменении.

## Главные выводы для переноса в LangGraph

1. Не переносить весь poller в graph как набор "умных агентов".
2. Оставить deterministic части deterministic.
3. Перенести orchestration из скрипта в:
- scheduler/job runner;
- один graph анализа заявки;
- один graph решения о доставке.
4. Сохранить изоляцию по `accountId` как отдельный системный инвариант.
5. Перенести state и delivery из файлов в `PostgreSQL`, но модель оставить похожей по смыслу.
6. Не переносить хранение полных данных заявок в новую систему: использовать их только transiently во время обработки.
