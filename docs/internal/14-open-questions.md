## Уже закрытые вопросы

Закрыто и больше не требует обсуждения на уровне MVP:
- `tenant` = отдельный клиент;
- на MVP `1 tenant = 1 user = 1 Telegram account`;
- `workspace` существует как отдельная сущность уже в MVP;
- `approval` делает сам пользователь;
- канал первого запуска только `Telegram`;
- на MVP только личный чат пользователя с ботом;
- provider layer должен быть расширяемым под будущие message providers;
- главный сценарий мониторинга: изменения по заявкам;
- основной тип источника для мониторинга: авторизованный кабинет / JSON API за логином;
- operational unit для `eds_monitor`: `accountId`;
- полные данные заявки на нашей стороне не хранятся;
- полная карточка заявки используется только transiently во время обработки;
- храним только признаки изменений и технический state;
- память полностью контролируется пользователем;
- система не может менять память без явного согласия пользователя;
- на MVP используется один ключ шифрования на среду;
- секреты на MVP живут в `env`;
- позже вероятен отдельный secrets layer уровня `Vault`;
- checkpoints LangGraph: стандартный Postgres-backed checkpointer;
- `LangSmith` в production на MVP не используется;
- администрирование на MVP только вручную внутренней командой;
- внутренняя команда может видеть переписку, но не PII;
- on-prem = managed single-tenant deployment;
- on-prem обновляется только после подтверждения клиента;
- on-prem должен поддерживать и локальную модель, и внешний API;
- отдельный операторский контур на MVP не нужен.
- integration layer `eds_monitor` строится сразу на `Python`;
- смешанный стек для MVP не используется.

## Оставшиеся реально открытые вопросы

## 1. Модель workspace после MVP

На MVP `workspace` есть как отдельная сущность, но фактически близок к модели "один клиент - одно пространство".

Открытый вопрос:
- как именно будет выглядеть расширение до связанного пространства из нескольких клиентов;
- какая модель membership потребуется;
- кто будет владельцем shared context и shared chat.

## 2. Формальная support-политика

На MVP зафиксировано:
- команда видит переписку;
- команда не видит PII.

Но пока открыто:
- как именно технически обеспечивать это разделение;
- какие служебные действия должны быть доступны команде;
- как оформлять аудит таких доступов.

## 3. Первый on-prem delivery profile

Пока зафиксирован общий принцип:
- managed deployment;
- master SSH key;
- обновления с подтверждением клиента.

Но остается открытым конкретный delivery format:
- `Docker Compose`;
- ansible-like automation;
- install agent;
- образ VM;
- гибридная схема.

## 4. Модель локального LLM для on-prem

Зафиксировано, что on-prem должен поддерживать:
- локальную модель;
- внешний API.

Открыто:
- какой локальный inference stack считать целевым первым;
- `vLLM`, `Ollama`, другой gateway;
- будет ли единый model adapter layer уже в первой реализации.

## 5. Минимальный состав internal API

Пока понятно, что публичной админки не будет.

Но все еще нужно определить:
- какие internal endpoints реально нужны команде в первом релизе;
- нужен ли read-only доступ к source runs, jobs, threads, approvals;
- что можно оставить только на уровне CLI/SQL.

## Практический вывод

Сейчас уже сняты почти все вопросы, которые мешают начать код.

Оставшиеся вопросы не блокируют старт skeleton и базовой реализации, но их стоит зафиксировать перед:
- production pilot;
- расширением workspace model;
- on-prem package;
- развитием `eds_monitor` в текущем runtime `Среды`.
