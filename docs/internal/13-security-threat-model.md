# Security Threat Model

## Цель

Зафиксировать базовую модель угроз для MVP, чтобы архитектура с самого начала учитывала:
- персональные данные;
- tenant isolation;
- внешние каналы;
- LLM/tool execution;
- on-prem перспективу.

Это не полный аудит безопасности, а рабочий стартовый документ.

## Главные активы

Что защищаем:
- персональные данные пользователей;
- профили и summary;
- токены Telegram и будущих интеграций;
- tenant/workspace boundaries;
- история диалогов;
- source-monitoring данные;
- approval decisions;
- ключи шифрования;
- системные логи и аудит.

## Главные поверхности атаки

### 1. Входящие webhook'и

Риски:
- поддельные запросы;
- replay;
- флуд;
- попытка сломать парсер update.

Защита:
- проверка bot binding;
- idempotency по external message id;
- rate limiting;
- строгая схема входного payload;
- аудит входящих webhook событий.

## 2. Внутренний API

Риски:
- подмена tenant/workspace контекста;
- доступ к чужим данным;
- неавторизованный approve/reject.

Защита:
- серверная валидация tenant/workspace;
- deny-by-default;
- обязательный audit;
- проверка actor identity;
- RLS в `PostgreSQL`.

## 3. Runtime state и память агента

Риски:
- смешение контекста разных пользователей;
- утечка PII в prompt/state;
- неконтролируемое накопление чувствительных данных в summaries.

Защита:
- отдельные thread и namespace;
- policy guard перед доступом к sensitive данным;
- не хранить сырой PII в graph state;
- ограниченный формат profile summary;
- явная очистка и редактирование памяти.

## 4. Tool use

Риски:
- LLM вызывает tool с опасным действием;
- tool получает лишние данные;
- tool пишет за пределы своего контекста;
- external side effects без согласования.

Защита:
- registry tools с уровнями риска;
- отдельная policy check перед tool call;
- read/write/sensitive classification;
- approval через `interrupt`;
- аудит каждого tool invocation.

## 5. Source monitoring

Риски:
- вредоносный контент на странице;
- prompt injection в тексте сайта;
- runaway scraping;
- повторная отправка мусора в канал.

Защита:
- parsing и extraction делать детерминированным кодом;
- LLM давать только очищенный и ограниченный input;
- запрещать модели интерпретировать HTML как команды;
- dedup;
- лимиты на частоту запуска и доставку.

## 6. Хранение данных

Риски:
- чтение PII из БД;
- дамп БД раскрывает профили;
- смешение runtime и secure данных;
- неправильные backup-практики.

Защита:
- field-level encryption;
- отдельные secure tables;
- минимум прав на чтение;
- шифрование backup;
- контроль доступа к дампам и admin-контуру.

## 7. Секреты и ключи

Риски:
- утечка bot token;
- утечка ключа шифрования;
- хранение секретов в коде или `.env`, попадающем в бэкапы/репозиторий.

Защита:
- секреты только во внешней secret-среде;
- ротация ключей;
- versioning ключей;
- раздельные ключи по средам;
- в будущем per-tenant keys.

## 8. Delivery в Telegram

Риски:
- отправка не тому чату;
- повторная отправка;
- утечка данных в чужой канал;
- массовый спам из-за багов scheduler.

Защита:
- outbox pattern;
- точная привязка `channel_binding`;
- dedup при отправке;
- лимиты на количество сообщений;
- audit каждой исходящей отправки.

## 9. On-prem инсталляции

Риски:
- ослабленные настройки по умолчанию;
- общий код, но разные ожидания по безопасности;
- локальный администратор получает доступ ко всему.

Защита:
- secure-by-default deployment profile;
- отдельные ключи на инсталляцию;
- отдельная документация hardening;
- отключение ненужных внешних интеграций;
- прозрачное разделение ответственности между системой и администратором клиента.

## Основные сценарии отказа, которые надо предотвратить

### Критичные

- один tenant видит данные другого tenant;
- ассистент отправляет PII не в тот чат;
- scheduler массово дублирует сообщения;
- approval обходится без подтверждения;
- токены и ключи оказываются в логах;
- runtime state содержит необработанный чувствительный профиль.

## Минимальные обязательные меры для MVP

- `RLS` в Postgres;
- field-level encryption;
- outbox pattern;
- idempotency для webhook и outbound;
- approvals для risky actions;
- audit log для чувствительных операций;
- policy guard перед tool access;
- логическая изоляция product data и secure data.

## Что нужно проверить перед первым пилотом

- нельзя ли получить чужой `thread`;
- нельзя ли подменить `tenant_id` в API;
- нельзя ли прочитать PII через LLM ответ;
- нельзя ли обойти approval;
- не дублирует ли scheduler задачи;
- не уходят ли секреты в логи;
- корректно ли работает tenant isolation в backup и restore процессе.
