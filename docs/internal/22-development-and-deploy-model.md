# Development And Deploy Model

## Цель

Зафиксировать, как ведется разработка `Sreda`, если:
- основная машина разработки находится локально;
- итоговая система должна работать на хосте `Sreda`;
- к хосту `Sreda` есть доступ по `SSH`.

## Базовая модель

- основная разработка кода ведется на текущей машине;
- хост `Sreda` используется как целевой runtime host;
- интеграционные проверки выполняются на хосте `Sreda`;
- выкладка идет через автоматизированный deploy по `SSH`.

## Почему именно так

Это дает нормальный баланс:
- локально удобнее писать код и быстро менять структуру;
- на хосте `Sreda` проверяется реальная среда выполнения;
- доступ к сайту и браузерная автоматизация остаются там, где они должны работать;
- production не превращается в основную dev-машину.

## Контуры

## 1. Local Development

Что происходит локально:
- пишется код;
- обновляется документация;
- запускаются unit-тесты;
- запускаются dry-run сценарии без реальных production secrets;
- проверяется базовая бизнес-логика.

Что не должно происходить локально:
- хранение production secrets;
- ручная production-эксплуатация;
- единственный источник истины по runtime state.

## 2. Remote Integration On Sreda Host

Что происходит на хосте `Sreda`:
- проверка login/session flow;
- проверка работы `eds_monitor` integration;
- проверка реального `Telegram` delivery;
- проверка runtime в среде, близкой к production;
- smoke-тест после deploy.

Это промежуточный контур между локальной разработкой и production.

## 3. Production On Sreda Host

Хост `Sreda` в итоговой модели:
- основной runtime host;
- место, где живут production secrets;
- место, где идет реальный polling и реальные уведомления.

## Правило разработки

Нельзя вести основную разработку руками прямо на хосте `Sreda`.

Причины:
- теряется контроль над изменениями;
- растет шанс сломать production;
- сложнее воспроизводить состояние;
- локальный контур разработки становится бесполезным.

Хост `Sreda` должен быть:
- integration host;
- deploy target;
- production host.

Но не основной authoring environment.

## Способ доставки кода

Рекомендуемый путь:
- код изменяется локально;
- затем отправляется на хост `Sreda` через deploy script по `SSH`.

Варианты технически:
- `git pull` на хосте `Sreda`, если там будет git checkout;
- `rsync`/`scp`, если хочется более прямой sync;
- гибридный вариант с архивом сборки.

Для MVP допустим любой из них, но важно:
- deploy должен быть воспроизводимым;
- deploy не должен зависеть от ручного копирования файлов по одному;
- production secrets не должны перетираться кодовой выкладкой.

## Рекомендуемая структура на хосте Sreda

Например:

- `~/apps/sreda/current`
- `~/apps/sreda/shared`
- `~/apps/sreda/releases/<timestamp>`

Где:
- `shared` содержит env, secrets, runtime data, logs;
- `releases` содержит выкладываемые версии;
- `current` указывает на активный релиз.

Даже если на MVP не делать полноценный release management, к такой схеме лучше тянуться сразу.

## Secrets

Секреты:
- не хранятся в кодовой базе;
- не приезжают из локальной машины при обычном deploy;
- лежат отдельно на хосте `Sreda`;
- подставляются через окружение или отдельный runtime env file.

## Runtime data

На хосте `Sreda` отдельно от кода должны жить:
- `.env` и runtime secrets;
- session storage;
- логи;
- база данных или путь до нее;
- любые persistent runtime directories.

Это нужно, чтобы deploy не ломал состояние системы.

## Режим выкладки

На MVP разумно вести такой процесс:

1. локально подготовить изменения;
2. локально прогнать базовую проверку;
3. сделать deploy на хост `Sreda`;
4. на хосте `Sreda` прогнать smoke-check;
5. только потом включать production flow.

## Что должен уметь deploy script

Минимально:
- подключаться по `SSH`;
- синхронизировать код;
- ставить или обновлять зависимости;
- применять миграции;
- перезапускать сервис;
- делать простую health-проверку.

Позже можно добавить:
- rollback;
- release directories;
- zero-downtime pattern;
- version pinning.

## Что нужно разделить сразу

Даже на одном хосте `Sreda` надо разделять:
- test/dev secrets;
- production secrets;
- test chats;
- production chats;
- test polling jobs;
- production polling jobs.

Иначе очень легко начать слать тестовые сообщения в рабочий канал.

## Практический итог

Согласованная модель такая:
- код разрабатывается на текущей машине;
- хост `Sreda` это integration/prod host;
- доставка кода идет по `SSH`;
- production state и secrets живут отдельно от кода;
- локальная машина не заменяет production;
- хост `Sreda` не используется как основная dev-машина.
