# Decisions From Open Questions

## Цель

Зафиксировать решения, которые были приняты при последовательном разборе исходного списка open questions.

Этот документ дополняет:
- [14-open-questions.md](./14-open-questions.md)
- [15-decisions.md](./15-decisions.md)

## 1. Tenant model

Решение:
- `tenant` на MVP это отдельный клиент.

Уточнение:
- один `tenant` = один пользователь;
- один `tenant` = один Telegram-аккаунт;
- все клиенты взаимодействуют через одного общего Telegram-бота.

## 2. Workspace model

Решение:
- `workspace` остается отдельной сущностью уже в MVP.

Почему:
- нужен задел под будущее объединение клиентов в общее пространство;
- позже могут появиться shared chats и связанное пространство.

## 3. Approval owner

Решение:
- все approvals на MVP подтверждает сам пользователь.

## 4. Channel scope

Решение:
- на MVP только `Telegram`;
- конкретного второго провайдера сейчас не реализуем;
- архитектура должна иметь message provider abstraction.

## 5. Telegram chat scope

Решение:
- на MVP только личный чат пользователя с ботом;
- без групп и каналов.

## 6. Monitoring scenario

Решение:
- главный мониторинговый сценарий на первом запуске это изменения по заявкам.

Уточнение:
- не только новые заявки;
- также важны изменения статуса, ответа, состояния ожидания действия пользователя.

## 7. Primary source type

Решение:
- базовый источник для мониторинга это авторизованный кабинет с JSON API за логином;
- operational unit для мониторинга это `accountId`.

## 8. Memory control

Решение:
- память полностью контролирует пользователь;
- система не может редактировать память молча;
- если система хочет что-то записать или поменять, она обязана явно спросить пользователя.

## 9. Claim storage policy

Решение:
- полные данные заявки на нашей стороне не хранятся;
- не храним описание, ответы и полную карточку;
- храним только признаки изменений и технический state.

Уточнение:
- полная заявка используется только transiently во время обработки;
- после анализа и отправки пользователю она не сохраняется в нашей БД.

## 10. Encryption key strategy

Решение:
- на MVP один ключ шифрования на среду;
- `per-tenant` и `per-workspace` ключи позже.

## 11. Secrets strategy

Решение:
- на MVP секреты живут в `env`;
- позже вероятен отдельный сервис секретов, скорее всего уровня `Vault`.

## 12. LangGraph checkpointing

Решение:
- на MVP используем стандартный Postgres-backed checkpointer LangGraph.

Уточнение:
- для `eds_monitor` flow полные payload заявок не должны сохраняться в persistent checkpoints.

## 13. Observability

Решение:
- `LangSmith` в production на MVP не используем;
- базовая наблюдаемость строится на логах, audit trail и внутренних технических логах.

## 14. MVP administration

Решение:
- на MVP администрирование только вручную внутренней командой;
- отдельная админка и операторский контур позже.

## 15. Support visibility

Решение:
- внутренняя команда может видеть переписку;
- внутренняя команда не должна видеть PII.

## 16. On-prem model

Решение:
- on-prem это managed single-tenant deployment;
- команда имеет доступ по master SSH key;
- система должна настраиваться автоматически;
- обновления должны подтягиваться с сервера обновлений, но ставиться только после подтверждения клиента или администратора клиента.

## 17. On-prem LLM mode

Решение:
- on-prem должен поддерживать оба режима:
  - локальный LLM;
  - внешний API провайдер.

## 18. EDS monitor integration stack

Решение:
- integration layer `eds_monitor` сразу переписывается на `Python`.

Почему:
- нужен единый стек `Python + LangGraph + FastAPI + PostgreSQL`;
- интеграции и runtime остаются в одном языке.

## 19. Feature model

Решение:
- дополнительные возможности системы управляются на уровне клиента через feature flags;
- первая такая feature: `eds_monitor`.

Уточнение:
- бесплатный режим работает без `eds_monitor`;
- после включения feature клиент получает полный monitoring-сценарий;
- на MVP включение делается вручную, без автоматизации оплаты.

## 20. Internal roles on MVP

Решение:
- на MVP отдельные внутренние роли не вводятся;
- пока только ручной доступ внутренней команды.

## Практический итог

Эти решения уже достаточны, чтобы:
- начать кодовый skeleton;
- поднимать core data model;
- строить Telegram-first MVP;
- развивать `eds_monitor` внутри текущей архитектуры `Среды`.
