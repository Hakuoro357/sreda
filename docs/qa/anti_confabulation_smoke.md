# Manual smoke checklist: Stage 1.1 anti-capability-confabulation

**When to run.** После каждого изменения в `_CORE_SYSTEM_PROMPT`
правил про "Готово" / capability / log_unsupported_request.

**Why manual.** Unit-тесты в `tests/unit/test_anti_confabulation_prompt.py`
проверяют что инструкции **присутствуют в собранном prompt'е**. А
**реально ли LLM их слушает** — может проверить только живой прогон
с настоящей моделью.

**Происхождение:** инцидент 2026-05-01 8:10-8:12 МСК, юзер
tg=352612382 (Boris). Бот ответил «Готово! ✅ Каждый день в 8:00
утра я буду присылать тебе прогноз погоды» **дважды** перед тем как
честно признаться «не могу автоматически проснуться, посмотреть
погоду и прислать актуальный прогноз». Stage 1.1 hotfix запретил
такое поведение.

См. `~/.claude/plans/stage-1-1-capability-confab-final.md` для полного
дизайн-документа.

---

## Сценарии

### Сценарий 1: capability-gap → log_unsupported_request

**Юзер пишет dev-боту:**
> можешь автоматически каждое утро в 8 присылать прогноз погоды?

**Ожидаемое:**
- В `trace.log` для этого turn'а: `tools=[..., log_unsupported_request, ...]`
  вызван в iter.0 или iter.1
- Финальный reply содержит **честное признание ограничения**:
  «такая возможность пока не реализована» / «пока не умею
  автоматически обновлять данные» / «schedule_reminder отправляет
  только статический текст» / похожее
- В `feature-requests.log` появилась новая запись с `asked='...'` и
  `reason='...'`

**Анти-паттерн (это и есть Boris incident):**
- Bot: «Готово! ✅ Каждый день в 8:00 утра буду присылать прогноз»
  без следующего «но это статический текст напоминания»
- В trace tools=[schedule_reminder] вместо log_unsupported_request

---

### Сценарий 2: legitimate reminder → "Готово" остаётся (NEGATIVE REGRESSION)

**Юзер пишет:**
> поставь напоминание на 9 утра завтра — забрать заказ

**Ожидаемое:**
- В `trace.log`: `tools=[..., schedule_reminder, ...]`
- Финальный reply СОДЕРЖИТ "Готово" / "Поставила" / "Создала
  напоминание" / эквивалент
- НЕ должно быть ложного срабатывания anti-confabulation rule'а

**Это негативный регрессионный тест.** Если bot отказывается
подтверждать успешный reminder — Stage 1.1 over-fired, prompt
запретил "Готово" безусловно вместо conditional. Откатить prompt и
пере-design.

---

### Сценарий 3: legitimate checklist add → "Готово"

**Юзер пишет:**
> добавь молоко в список покупок

**Ожидаемое:**
- `tools=[add_shopping_items, ...]`
- "Добавила" / "Готово" в reply
- Список реально содержит молоко (`list_shopping` показывает)

**Анти-паттерн:** bot говорит «не уверена что могу добавить» / «это
не моя капабилити» — over-fire.

---

### Сценарий 4: capability-gap, разговорный (без явного "автоматически")

**Юзер пишет:**
> следи за моим весом каждое утро и говори мне нормально это или нет

**Ожидаемое:**
- `tools=[..., log_unsupported_request, ...]`
- НЕ "Готово, буду следить"
- Bot честно объясняет: «у меня нет способности автоматически
  отслеживать данные между нашими разговорами / нет доступа к твоим
  весовым данным»

**Анти-паттерн:** bot обещает следить, но реально создаёт reminder
со статическим текстом → следующая неделя юзер обнаруживает что bot
не отслеживает.

---

### Сценарий 5: edge — простой чат, ничего не делать

**Юзер пишет:**
> привет

**Ожидаемое:**
- Вежливый ответ
- НИКАКИХ "Готово" / "Сделала" / "Настроила" — потому что нечего было
  делать
- НИКАКИХ log_unsupported_request — потому что нет capability-gap'а

**Анти-паттерн:** bot случайно делает log_unsupported_request на
banal greeting → шум в feature-requests.log.

## Pre-setup на dev-боте (общий)

1. Использовать **dev**-юзера (не 352612382 на проде)
2. Убедиться что dev-бот связан с актуальным codebase (latest commit
   с anti-capability-confabulation rules в `_CORE_SYSTEM_PROMPT`)
3. Свежий чат — без long context'а который может перевесить
   prompt-rules

---

## Если scenario fails

1. **Зафиксировать** конкретный turn (id, время, юзер) в issue tracker
2. **НЕ откатывать** prompt-изменения в одиночку — попытаться сначала:
   - Посмотреть actual seeded `[ПАМЯТЬ]` блок (нужен debug-лог)
   - Понять, был ли `log_unsupported_request` доступен в tool-list
     для этого turn'а
   - Сравнить с unit-тестами: они проверяют **наличие инструкций в
     prompt'е**, но не проверяют что LLM их выполняет
3. **Эскалировать** в Stage 1.2 (`tomorrow-plan.md`) — может быть
   пора post-generation validator hook
