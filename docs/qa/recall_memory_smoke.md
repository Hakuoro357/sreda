# Manual smoke checklist: recall_memory proactive policy

**When to run.** После каждого изменения в:
- `src/sreda/runtime/tools.py::build_memory_tools::recall_memory` docstring
- `src/sreda/runtime/handlers.py::_CORE_SYSTEM_PROMPT` блок про память
- `src/sreda/runtime/graph.py::node_load_memories` (pre-retrieval)

**Why manual.** Unit-тесты в `tests/unit/test_recall_memory_prompt.py`
проверяют, что инструкции **присутствуют в собранном prompt'е**. А
**реально ли LLM их слушает** — может проверить только живой прогон с
настоящей моделью. Mock LLM возвращает то, что мы в него запишем —
никаких регрессий не ловит.

**Происхождение:** инцидент 30 апреля 2026 (юзер tg=755682022).
Бот ответил «пока только одна ткань: Лён хлопок пудра», когда в
`assistant_memories` было 5+ ткань-записей. См. `tomorrow-plan.md`
пункт 11 для root cause.

---

## Pre-setup (обязательно перед запуском)

1. На **dev-боте** (НЕ на боевом 352612382) залогиниться под
   тестовым юзером.
2. Через CLI / админку или вручную через API создать в
   `assistant_memories` 5 записей одной темы. Пример — «любимые
   блюда»:
   - «Любит борщ — особенно с пампушками и чесноком»
   - «Окрошку делает на квасе, не на кефире»
   - «Лазанью только мясную, без шпината»
   - «Сырники на завтрак раз в неделю»
   - «Плов готовит в казане, обязательно с зирой»
3. Плюс 5-10 **distractor** записей других тем (семья, расписание,
   ткани, что угодно НЕ про блюда). Это нужно чтобы pre-retrieval
   top-10 имел из чего отфильтровать блюда.
4. **Проверка корректности setup'а перед запуском checklist'а.**
   Прогнать тестовый pre-retrieval (через debug-endpoint, временный
   `logger.info` в `node_load_memories`, или ad-hoc psql + python script):
   - Запрос: «перечисли все мои любимые блюда»
   - Ожидание: seeded `[ПАМЯТЬ]` содержит **меньше 5** блюд (т.е.
     часть теряется в top-k cutoff)
   - Если все 5 попали — добавить ещё distractor memories ИЛИ
     временно снизить `top_k` на dev (например на 3-5), чтобы
     создать условие неполного seeded набора.
   - **Без этой проверки checklist не доказывает то что проверяет** —
     если seeded и так полный, бот ответит правильно даже без
     proactive recall.

## Сценарии

### 1. List-style query: бот делает proactive recall_memory

**Юзер пишет:** «перечисли все мои любимые блюда»

**Ожидаемое:**
- В `trace.log` для этого turn'а: `tools=[..., recall_memory, ...]`
  (recall_memory был вызван LLM'ом)
- В ответе юзеру — **все 5 блюд** (а не 2-3 из seeded `[ПАМЯТЬ]`)

**Почему это важно:** ровно тот кейс который сломался 30 апреля.
Если recall_memory не вызывается — proactive policy не работает.

---

### 2. «Нет данных»: recall_memory ПЕРЕД негативным ответом

**Юзер пишет:** «есть ли у тебя данные про макароны?»
(тема которой точно нет в БД и не похожа на блюда)

**Ожидаемое:**
- В `trace.log`: `tools=[..., recall_memory, ...]` — бот **попробовал**
  поискать перед тем как отвечать «нет»
- В ответе: что-то вроде «не нашла записей про макароны, рассказать
  если хочешь?» — но **только после** recall_memory

**Анти-паттерн:** бот сразу отвечает «у меня нет данных по
макаронам, ты не записывала» без вызова recall_memory.

---

### 3. Анти-internals: бот не объясняет внутреннюю кухню

**Юзер пишет:** «откуда ты это знаешь?» (после успешного ответа из
сценария 1)

**Ожидаемое:** ответ из категории «из памяти, ты записывала
раньше» / «ты говорила об этом ранее».

**Анти-паттерн (любое из этих слов = провал):**
- «выборка»
- «контекст сессии» / «окно контекста»
- «релевантность» / «top-k»
- «retrieval» / «retrieval-поиск»
- «embedding» / «семантический поиск»
- «передаётся ограниченное количество записей»

**Почему важно:** 1 мая 11:57 МСК бот написал юзеру именно
«передаётся ограниченное количество самых релевантных записей» —
это leak реализации, юзер не должен этого видеть.

---

### 4. Анти-confabulation: бот не выдумывает свои действия

**Юзер пишет:** «почему ты сначала ответила X а потом Y?»
(где X и Y — реальные предыдущие ответы бота из этой сессии)

**Ожидаемое:** ответ из категории «не отслеживаю свои предыдущие
шаги в этой сессии настолько точно, могу проверить заново если
нужно».

**Анти-паттерн:** бот описывает выдуманную последовательность вида
«я сначала посмотрела в [ПАМЯТЬ], увидела только две записи, потом
ты сказала "в переписке было много", и я сделала recall_memory».
Если этой последовательности **в actual trace не было** — это
hallucination. Реальный пример 1 мая 11:56 — бот выдумал именно
такую цепочку.

**Как проверить:** trace конкретного turn'а от 11:56 не содержит
шага «recall_memory called после реплики юзера», но в outbox-ответе
бот описывает именно такой шаг.

---

### 5. Голосовой list-query: тот же proactive recall

**Юзер пишет голосом:** «помнишь, я говорила про любимые блюда?»

**Ожидаемое:** в `trace.log`:
- `voice.transcribe` ok
- В iter.0 либо iter.1 — `tools=[..., recall_memory, ...]`
- Финальный ответ содержит хотя бы 3-5 блюд из памяти

**Почему отдельный сценарий:** проверяем что voice-pipeline не
теряет proactive policy (в прошлом для voice бывали отдельные
prompt-сборки — нужно убедиться что эта тоже включает обновлённые
инструкции).

## Проверка trace'а

После каждого сценария — посмотреть `/var/log/sreda/trace.log` на
проде ИЛИ локальный trace при dev-прогоне:

```
=== TRACE trace_<id> 2026-... user=user_tg_<dev_user_id> chan=telegram ===
   webhook.received       type=text
   ack.sent               ...
   llm.iter.0             ... tools=[recall_memory]   ← должно быть
   llm.iter.1             ... tools=[]
   chat.reply             ...
   outbox.delivered       ...
```

Ключевая строка: `tools=[recall_memory]` (или среди других tools).
Если она отсутствует во всех iter.0..N — recall_memory не вызывался.

## Если scenario fails

1. **Зафиксировать** конкретный turn (id, время, юзер) в issue tracker
2. **НЕ откатывать** prompt-изменения в одиночку — попытаться сначала:
   - Посмотреть actual seeded `[ПАМЯТЬ]` блок (нужен debug-лог в `node_load_memories`)
   - Понять, был ли recall_memory доступен в tool-list для этого turn'а
   - Сравнить с unit-тестами: они проверяют **наличие инструкций в
     prompt'е**, но не проверяют что LLM их выполняет
3. **Эскалировать** в пункт 11.2 (`tomorrow-plan.md`) — Stage 2
   observability нужно добавить раньше планируемого
