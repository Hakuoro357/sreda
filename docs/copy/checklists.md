# Копирайт + UX: Чек-листы (списки дел)

Документ для команды и для агента. Описывает третий тип «списков»
в продукте — `Checklist` + `ChecklistItem`. Сценарии триггеров,
LLM-тулы, тексты пользователю, Mini App экран.

---

## 0. Зачем

В неделю использования (2026-04-21..25) обнаружилась дыра в продукте:
швея надиктовывала «план кроя» из 7 пунктов разного материала. LLM
выбирала наименее плохой инструмент — `add_task` ×3 — и расцеплял
пункты по разным датам в Mini App «Расписание». Юзер увидел и сказал:

> Это не расписание, а список дел с галочками как в покупках.

Корень: в системе было ДВА типа «списков» — Task (с датой) и
ShoppingListItem (продукты), а произвольные «дела со статусом сделано»
ложиться было некуда.

**Чек-листы закрывают эту дыру.** Это произвольные именованные списки
без дат, с галочкой `pending/done` на каждом пункте.

---

## 1. Когда что использовать (для LLM и для команды)

В системе ТРИ типа «списков». Правило в `_HOUSEWIFE_FOOD_PROMPT`:

| Тип | Тул | Когда |
|---|---|---|
| **Продукты в магазин** | `add_shopping_items` | «купить молоко, хлеб, яйца», «добавь в список покупок X» |
| **События с датой/временем** | `add_task` | «встреча завтра в 10», «дедлайн до 25 апреля», «кружок в понедельник 9:00» |
| **Произвольный план без дат** | `create_checklist` + `add_checklist_items` | «план кроя на эту неделю», «дела на дачу», «чек-лист сборов в школу», «материалы для ремонта» |

**Главное правило:** если пунктов несколько и нет конкретных дат —
это checklist. НЕ размазывай пункты по дням расписания через add_task.
LLM-инструкция в prompt'е содержит явный pointer на эту ошибку.

## 2. LLM tools (6 штук)

Все определены в `src/sreda/services/housewife_chat_tools.py`,
обёрнуты в `@lc_tool`, возвращают компактные строки для LLM
(паттерн `ok:<verb>:<id>` или `error:<reason>`).

### `create_checklist(title)`

Новый именованный список (status=active). Пара с `add_checklist_items`
следующим вызовом для наполнения.

**Возврат:** `ok:created:checklist_<id>:<title>` / `error:<msg>`

### `add_checklist_items(list_id_or_title, items)`

Пакетно добавляет пункты в конец указанного списка. Резолвит
`list_id_or_title` либо по точному id (`checklist_xxx`), либо по
fuzzy substring к title активного списка.

**Особенность:** если список не найден — **создаёт новый** с этим
title. Это упрощает LLM-flow: один вызов вместо двух.

**Возврат:** `ok:added:N:list=<id>` / `error:<msg>`

### `list_checklists()`

Все active со счётчиками `N pending, N done, N total`.

**Возврат (multi-line):**
```
[checklist_xxx] · План кроя · 5 pending, 2 done, 7 total
[checklist_yyy] · Дела на дачу · 4 pending, 0 done, 4 total
```
Или `no checklists`.

### `show_checklist(list_id_or_title)`

Содержимое одного списка с галочками.

**Возврат (multi-line):**
```
# План кроя (checklist_xxx)
[clitem_a1] ☐ Лаванда 298 ТС, простыня 141×200×19
[clitem_a2] ☑ Шампань страйп, простыня 202×204×26
[clitem_a3] ☐ Лайм, простыня 140×200×20
```

Используется когда юзер просит «покажи план», «что осталось»,
«что я ещё не сделал».

### `mark_checklist_item_done(list_id_or_title, item_title_match)`

Найти pending пункт по подстроке, поставить done. Если в pending не
нашлось — fallback по всем (включая cancelled).

**Возврат:** `ok:done:<item_id>:<title>` / `error:item_not_found:<needle>`

### `archive_checklist(list_id_or_title)`

Спрятать список из active (не удалить). Status='archived'.

**Возврат:** `ok:archived:<id>` / `error:not_found`

## 3. Текстовые сценарии

Чек-листы реактивные (LLM сам решает что вызывать на основе фразы
юзера) — никаких хардкодных welcome-фраз тут нет, всё рисует LLM.
Но в pending-bot есть демо-пример (`pending_bot.py::_BRANCH_WHAT`),
который намекает что бот это умеет.

### Сценарий A — создание списка с нуля

```
Юзер (голос): «Запиши план кроя на эту неделю: лаванда 298 простыня
        141 на 200, шампань 202 на 204, лайм 140 на 200, …»

LLM tool: create_checklist(title="План кроя на эту неделю")
LLM tool: add_checklist_items(
    list_id_or_title="План кроя на эту неделю",
    items=[
      "Лаванда 298 ТС, простыня 141×200×19",
      "Шампань страйп, простыня 202×204×26",
      "Лайм, простыня 140×200×20",
    ],
)
LLM ответ: «Создала список «План кроя на эту неделю» с 7 пунктами ✅»
```

### Сценарий B — добавление в существующий

```
Юзер: «Добавь в план кроя ещё бежево-розовый 213ТС, пододеяльник 150×200»

LLM tool: add_checklist_items(
    list_id_or_title="план кроя",  # fuzzy match
    items=["Бежево-розовый 213 ТС, пододеяльник 150×200"],
)
LLM ответ: «Добавила в «План кроя на эту неделю» ✅»
```

### Сценарий C — отметить сделанное

```
Юзер: «Закройила лаванду»

LLM tool: list_checklists()                 # понять в каком списке
LLM tool: mark_checklist_item_done(
    list_id_or_title="План кроя",
    item_title_match="лаванда",
)
LLM ответ: «Отметила: Лаванда 298 ТС ☑»
```

### Сценарий D — recall

```
Юзер: «Покажи план кроя»

LLM tool: show_checklist("план кроя")
LLM ответ: «Вот что в плане:
☐ Лаванда 298 ТС, простыня 141×200×19
☑ Шампань страйп, простыня 202×204×26
☐ Лайм, простыня 140×200×20
…»
```

LLM ОБЯЗАНА вызвать `show_checklist` — НЕ галлюцинировать ответ из
памяти AI. Это явно записано в prompt'e.

### Сценарий E — закрыть список

```
Юзер: «Уберём этот план кроя, он уже не актуален»

LLM tool: archive_checklist("план кроя")
LLM ответ: «Архивировала «План кроя» — больше не показываю в списках. Если вернётесь — скажите, восстановлю.»
```

## 4. Mini App — экран `#/checklists`

Read-only в MVP — мутации только голосом / текстом.

### Home-card (на дашборде)

- icon `📋`, title «Списки»
- subtitle: `N открытых в M списках` (если 0 списков — карточка скрыта)
- count → бейдж с N открытых

### Экран

```
📋 Списки

▾ План кроя на эту неделю              5 / 7
   ☐ Лаванда 298 ТС, простыня 141×200×19
   ☑ Шампань страйп, простыня 202×204×26  ← strikethrough
   ☐ Лайм, простыня 140×200×20
   ...

▾ Дела на дачу                         4 / 4
   ☐ Привезти бензин для генератора
   ☐ Проверить колодец
   ...

Чтобы добавить или отметить пункт — скажи мне голосом
(«закройила лаванду», «добавь в план кроя X»).
Архивировать список — «закрой план кроя».
```

- Каждый список — `<details>` (раскрывается).
  - **open по умолчанию** если есть pending пункты.
  - **collapsed** если все done.
- ☐/☑/✗ — visual-only. Тап **не меняет** статус.
- Footer-подсказка про голос.

API endpoint: `GET /api/v1/checklists` — возвращает все active с
вложенными items.

## 5. Анти-сталкер фильтры

Чек-листы создаются ТОЛЬКО по явному запросу юзера. Нет проактивных
триггеров — не как aha-moments.

- ✅ LLM не «угадывает» что юзер хочет завести чек-лист (только когда
  юзер сам диктует план).
- ✅ В Mini App нет «smart-suggest» создать чек-лист.
- ✅ Read-only Mini App: бот не может «что-то скрыть» от юзера —
  всё что в списке, юзер сам туда положил.
