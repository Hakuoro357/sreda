# Копирайт: приветственные реплики

Документ описывает ВСЕ тексты, которые бот отправляет новому юзеру в
фазе до и сразу после одобрения админом. Это source of truth для
команды — при любом вопросе «что у нас там в welcome» смотри сюда.

Каждый текст сопровождается:
- цитатой (как видит юзер)
- триггером (когда отправляется)
- кнопками (если есть)
- местом в коде (файл:функция)

---

## 0. Контекст

Новый пользователь присылает `/start`. В БД создаётся `Tenant` с
`approved_at = NULL` («pending approval»). Админ должен вручную
нажать «Одобрить» в `/admin/users` — это «костыль до запуска
подписок» (защита от массового набега).

Юзер в pending-фазе получает **pending-bot** (scripted, без LLM).
С 2026-04-27 (вечер) это **цепочка из 10 коротких сообщений** —
одно intro + 8 функциональных блоков с кнопкой «следующая тема» +
финальное сообщение про бета-тест без кнопок.

История версий:
* 2026-04-25: 7 веток демо-рассказа с кнопками-темами.
* 2026-04-27 (утро): упрощено до одного длинного welcome без кнопок.
* 2026-04-27 (вечер): разбито обратно на 10 коротких — длинная
  портянка плохо читалась в Telegram.

Когда админ одобряет — приходит housewife-ориентированный welcome
с объяснением «зачем» и эскейп-кнопкой «Может, позже» (см. секцию 2).

**Ключевые принципы тона** (не обсуждается, прошито в prompt + тестах):

1. **Анти-сталкер.** Не пишем первыми без повода, не считаем вслух
   упоминания, при запросе данных объясняем «зачем».
2. **Gender-neutral к юзеру.** Никакого прошедшего времени в
   ж.р. («ты сказала», «ты сама», «ты попросила»). Допустимы
   безличные формы или скобочные `(-а)`. Про Среду ж.р. разрешён.
3. **«Ты» форма** во всех текстах без исключений.
4. **Кнопки** на каждый вопрос. «Да / Нет» запрещены — только
   конкретные реплики («Да, собери меню», «Не сейчас»).
5. **Цены — плейсхолдеры** из БД через `services/pricing.py`.
6. **Без Markdown.** Telegram не рендерит `**жирный**`.
7. **Без технических терминов.** Никаких «агент», «Mini App»,
   «лимит тарифа», «хронические лекарства».

## 1. Pending welcome (цепочка из 11 сообщений)

**Триггер.** Любое входящее от юзера в pending-фазе:
- `/start` от непрошедшего approval юзера (`tenant.approved_at IS NULL`)
- любой text-message
- любое голосовое (без STT — pending-фаза не тратит Groq/Yandex)
- любой callback (включая устаревшие `pb:<branch>` от прошлых версий —
  fallback'ятся в intro)

**Файл.** `src/sreda/services/pending_bot.py`

**Поток.** Каждое сообщение, кроме closing, содержит ровно одну
кнопку «следующая тема →». Юзер тапает → приходит следующее
сообщение. Если юзер пропустит шаг и напишет text — вернётся в intro.

| # | branch_key | Кнопка ведёт на | Тема |
|---|---|---|---|
| 1 | `intro` | `🎙️ Голос →` | Представление + список доменов |
| 2 | `voice` | `📅 Расписание →` | Голос как основной режим работы |
| 3 | `schedule` | `🔔 Напоминания →` | Задачи + повторы + Mini App |
| 4 | `reminders` | `📝 Дела →` | Кнопки Сделал/Отложить, без auto-re-ping |
| 5 | `checklists` | `🛒 Покупки →` | Списки без дат |
| 6 | `shopping` | `📖 Рецепты →` | Голосовое добавление + категории |
| 7 | `recipes` | `👨‍👩‍👧 Семья →` | Книга + КБЖУ + поиск по ингредиентам |
| 8 | `family` | `🧠 Память →` | Состав семьи (без обещаний имён) |
| 9 | `memory` | `🚫 Чего не делаю →` | Произвольные факты «запомни, что Х» |
| 10 | `dont_do` | `Готово ✓` | Анти-сталкер декларация |
| 11 | `done` | — (нет кнопок) | Финал: «бета-тест, я выйду первой» |

**Тексты сообщений** — см. константы `_INTRO`, `_VOICE`, `_SCHEDULE`,
`_REMINDERS`, `_CHECKLISTS`, `_SHOPPING`, `_RECIPES`, `_FAMILY`,
`_MEMORY`, `_DONT_DO`, `_DONE` в `pending_bot.py`. Никаких внешних
копий — изменения текста делаются ТАМ, этот документ — индекс.

**Замечания / 152-ФЗ:**
- Из ветки `family` убраны диеты, аллергии, лекарства
  (спецкатегория ст. 10 152-ФЗ). Также убрано конкретное обещание
  «не «ребёнку», а «Маше»» — готовит UX к Части 2 (роли вместо
  имён). При попытке сохранить медицинский маркер privacy guard
  заменит его на `[allergy]`/`[diagnosis]` в LLM-prompt.
- В ветке `dont_do` явно декларируем что мед.данные не сохраняем
  в открытом виде — часть стратегии «не оператор ПДн».
- Голосовой блок (`voice`) — самое первое сообщение после intro:
  голос основной режим работы со Средой, а не «ещё одна возможность».
- Меню на неделю исключено: функционал закомментирован в продукте
  на 2026-04-27.
- Теги в рецептах в БД есть (`tags_json`), но в Mini App не
  показываются — упоминание убрано из текста ветки `recipes`.

**Closing message (`done`):** «На этом всё, что хотела рассказать
про себя. Сейчас Среда на закрытом бета-тестировании. Как только
модератор одобрит твой доступ — я сама выйду на контакт первой.
Делать ничего не нужно — просто жди сообщения. Обычно занимает
пару часов.» Без слов про «я спрошу две вещи как тебя зовут и
ты/вы» — это сюрприз для пост-approval flow.

**Aliases для backwards-compat.** Устаревшие callback_data из
прошлых версий (`pb:welcome`, `pb:what`, `pb:demo_morning`,
`pb:menu_example`, `pb:life`) маршрутизируются в `intro` — чтобы
юзер с зависшим Telegram-кешом не получил пустой ответ. ВНИМАНИЕ:
`pb:memory` теперь не alias, а активный branch (новый блок).

## 2. Welcome после approval (simplified, 2026-04-27 вечер)

**Триггер.** Админ нажимает «Одобрить» в `/admin/users`.

**Файл.** `src/sreda/services/onboarding.py::build_post_approve_message`

**Текст:**
```
✅ Готово! Модератор открыл доступ — рада знакомству.

Прежде чем приступим, подскажи, как к тебе обращаться? Имя или
ник, как удобно. Это нужно, чтобы напоминания и сообщения были
по-человечески, а не «уважаемый пользователь».
```

**Кнопки.** Нет (свободный ввод имени).

**Что было раньше и почему отказались.**

С 2026-04-25 по 2026-04-27 (день) post-approve flow был
3-ступенчатым: вопрос про имя → вопрос про ты/вы → housewife-welcome
про семью с кнопками. Свернули в одно сообщение, потому что:
- Юзер только что прошёл pending-цепочку из 11 сообщений (включая
  представление и обзор всех функций) — представление избыточно.
- 3 шага после длинной pending-цепочки — слишком много
  «настройки» перед первым реальным ответом.
- Расспросы про семью / диеты / аллергии плохо ложатся в первое
  сообщение — лучше в естественный диалог.

**Замечания.**
- LLM сама извлекает имя через `onboarding_answered(topic="addressing",
  summary="<имя>")` tool, когда юзер пишет ответ. Webhook не
  перехватывает первый text как display_name.
- Расспросы про семью / диеты / другие данные смещены в обычный
  chat-flow — LLM спрашивает когда уместно, по своему усмотрению
  (тон housewife-prompt'а уже задаёт правильный flow).
- Форма обращения «вы» удалена из flow и перенесена в backlog
  (`docs/tomorrow-plan.md` пункт 8). Колонка `address_form`
  остаётся в БД для возврата фичи.

**Sanitizer для display_name (2026-04-28 incident).** На проде
обнаружили 3 испорченных `display_name` — LLM передавала в
`onboarding_answered.summary` целые фразы вместо имени:
- `"Пользователя зовут Борис."`
- `"Пользователь хочет, чтобы его называли «Шеф»."`
- `"Пользователя зовут Повелитель — просил называть его так."`

Защита трёхслойная (см. `services/housewife_onboarding._extract_short_name`):
1. **Tool docstring** в `services/housewife_chat_tools.onboarding_answered` —
   явные ПРАВИЛЬНО / ЗАПРЕЩЕНО примеры.
2. **Prompt rule №5** в `format_for_prompt` — расширен с примерами
   запрещённых summary.
3. **Backend sanitizer** `_extract_short_name` — последняя линия
   обороны: режет префиксы «Пользователя зовут», «Меня зовут»,
   «Пользователь хочет, чтобы его называли», обрезает кавычки и
   терминальную пунктуацию, оставляет только первую часть до
   тире/запятой/точки. Применяется в `_set_display_name`
   (housewife) и в `runtime.handlers._validate_proposed_field`.

Существующие 3 испорченных значения исправлены вручную SQL UPDATE'ом
в проде:
- `tup_4da5c5567362487da11a22e7` (`tg_352612382`) → «Борис»
- `tup_d05ba443080c491694cd47f5` (`tg_30223042`) → «Повелитель»
- `tup_482305551f564a979e90b1b5` (`tg_981993906`) → «Шеф»
