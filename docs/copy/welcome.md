# Копирайт: приветственные реплики

Документ описывает ВСЕ тексты, которые бот отправляет новому юзеру в
фазе до и сразу после одобрения админом. Это source of truth для
команды — при любом вопросе «что у нас там в welcome» смотри сюда.

Каждый текст сопровождается:
- цитатой (как видит юзер)
- триггером (когда отправляется)
- кнопками (если есть)
- местом в коде (файл:функция)

---

## 0. Контекст

Новый пользователь присылает `/start`. В БД создаётся `Tenant` с
`approved_at = NULL` («pending approval»). Админ должен вручную
нажать «Одобрить» в `/admin/users` — это «костыль до запуска
подписок» (защита от массового набега).

Юзер в pending-фазе получает **pending-bot** (scripted, без LLM).
С 2026-04-27 (вечер) это **цепочка из 10 коротких сообщений** —
одно intro + 8 функциональных блоков с кнопкой «следующая тема» +
финальное сообщение про бета-тест без кнопок.

История версий:
* 2026-04-25: 7 веток демо-рассказа с кнопками-темами.
* 2026-04-27 (утро): упрощено до одного длинного welcome без кнопок.
* 2026-04-27 (вечер): разбито обратно на 10 коротких — длинная
  портянка плохо читалась в Telegram.

Когда админ одобряет — приходит housewife-ориентированный welcome
с объяснением «зачем» и эскейп-кнопкой «Может, позже» (см. секцию 2).

**Ключевые принципы тона** (не обсуждается, прошито в prompt + тестах):

1. **Анти-сталкер.** Не пишем первыми без повода, не считаем вслух
   упоминания, при запросе данных объясняем «зачем».
2. **Gender-neutral к юзеру.** Никакого прошедшего времени в
   ж.р. («ты сказала», «ты сама», «ты попросила»). Допустимы
   безличные формы или скобочные `(-а)`. Про Среду ж.р. разрешён.
3. **«Ты» форма** во всех текстах без исключений.
4. **Кнопки** на каждый вопрос. «Да / Нет» запрещены — только
   конкретные реплики («Да, собери меню», «Не сейчас»).
5. **Цены — плейсхолдеры** из БД через `services/pricing.py`.
6. **Без Markdown.** Telegram не рендерит `**жирный**`.
7. **Без технических терминов.** Никаких «агент», «Mini App»,
   «лимит тарифа», «хронические лекарства».

## 1. Pending welcome (цепочка из 10 сообщений)

**Триггер.** Любое входящее от юзера в pending-фазе:
- `/start` от непрошедшего approval юзера (`tenant.approved_at IS NULL`)
- любой text-message
- любое голосовое (без STT — pending-фаза не тратит Groq/Yandex)
- любой callback (включая устаревшие `pb:<branch>` от прошлых версий —
  fallback'ятся в intro)

**Файл.** `src/sreda/services/pending_bot.py`

**Поток.** Каждое сообщение, кроме closing, содержит ровно одну
кнопку «следующая тема →». Юзер тапает → приходит следующее
сообщение. Если юзер пропустит шаг и напишет text — вернётся в intro.

| # | branch_key | Кнопка ведёт на | Тема |
|---|---|---|---|
| 1 | `intro` | `🎙️ Голос →` | Представление + список доменов |
| 2 | `voice` | `📅 Расписание →` | Голос как основной режим работы |
| 3 | `schedule` | `🔔 Напоминания →` | Задачи + повторы + Mini App |
| 4 | `reminders` | `📝 Дела →` | Кнопки Сделал/Отложить, без auto-re-ping |
| 5 | `checklists` | `🛒 Покупки →` | Списки без дат |
| 6 | `shopping` | `📖 Рецепты →` | Голосовое добавление + категории |
| 7 | `recipes` | `👨‍👩‍👧 Семья →` | Книга + поиск по ингредиентам |
| 8 | `family` | `🚫 Чего не делаю →` | Имена/возрасты/кружки (без мед.данных) |
| 9 | `dont_do` | `Готово ✓` | Анти-сталкер декларация |
| 10 | `done` | — (нет кнопок) | Финал: «бета-тест, модератор подключит» |

**Тексты сообщений** — см. константы `_INTRO`, `_VOICE`, `_SCHEDULE`,
`_REMINDERS`, `_CHECKLISTS`, `_SHOPPING`, `_RECIPES`, `_FAMILY`,
`_DONT_DO`, `_DONE` в `pending_bot.py`. Никаких внешних копий —
изменения текста делаются ТАМ, этот документ — индекс.

**Замечания / 152-ФЗ:**
- Из ветки `family` убраны диеты, аллергии, лекарства
  (спецкатегория ст. 10 152-ФЗ). Оставлены preference-данные:
  имена, возрасты, кружки, любимые/нелюбимые блюда. При попытке
  сохранить медицинский маркер privacy guard заменит его на
  `[allergy]`/`[diagnosis]` в LLM-prompt.
- В ветке `dont_do` явно декларируем что мед.данные не сохраняем
  в открытом виде — часть стратегии «не оператор ПДн».
- Голосовой блок (`voice`) — самое первое сообщение после intro:
  голос основной режим работы со Средой, а не «ещё одна возможность».
- Меню на неделю исключено: функционал закомментирован в продукте
  на 2026-04-27.

**Aliases для backwards-compat.** Устаревшие callback_data из
прошлых версий (`pb:welcome`, `pb:what`, `pb:demo_morning`,
`pb:menu_example`, `pb:memory`, `pb:life`) маршрутизируются в `intro`
— чтобы юзер с зависшим Telegram-кешом не получил пустой ответ.

## 2. Welcome после approval

**Триггер.** Админ нажимает «Одобрить» в `/admin/users`.

**Файл.** `src/sreda/services/onboarding.py::build_welcome_message`

**Текст:**
```
✅ Готово! Модератор подключил.

Теперь я могу отвечать по-настоящему.

Чтобы я была полезной сразу — коротко расскажи про семью: имена и
возрасты (чтобы обращаться правильно), и есть ли у кого-то диета
или аллергия (чтобы меню подходило всем).

Ничего не обязательно — можно начать без этого и рассказывать по
мере того, как будет уместно.

Пиши голосом или текстом — как удобно.
```

**Кнопки:**
- `Может, позже` → btn_reply (переводит в свободный чат без формы)
- `Расскажу про семью` → btn_reply (LLM ведёт диалог сбора)

**Замечания.** Применены все Фильтры 1-6:
- Фильтр 1 (зачем): каждое «спроси X» объяснено в скобках.
- Фильтр 2 (progressive): просим минимум — имена + диеты. Остальное
  (лекарства, кружки, др.) всплывёт когда юзер сам скажет.
- Фильтр 3 (эскейп): кнопка «Может, позже» прямо предусмотрена.
- Фильтр 4 (без счёта): не применимо (нет данных ещё).
- Фильтр 5 (без спонтанных чек-инов): не применимо.
- Фильтр 6 (opt-in на память): сработает при первом `save_core_fact`.
