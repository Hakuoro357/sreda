# Копирайт: Aha-moments

Документ описывает ВСЕ проактивные «aha»-сообщения, которые бот шлёт
юзеру сам (без его прямого запроса). Это критическая граница
«помощник vs сталкер» — каждое сообщение бота-по-инициативе проходит
через whitelist и тон-фильтры.

Каждое aha-сообщение сопровождается:
- триггером (когда именно срабатывает)
- timing (во сколько + задержка)
- текстом (цитата)
- кнопками (опции юзера)
- местом в коде
- статусом реализации (✅ | ⚠️ | 🗑️)

---

## 0. Контекст

Проактивность — ключевое УТП Среды («сам напомнил, не надо
проверять»). Но без границ превращается в сталкинг. Принцип из
плана v2:

> Бот МОЖЕТ инициировать сообщения сам. Но каждый новый повод для
> проактивного контакта согласуется с пользователем и добавляется в
> **явный whitelist**. Нельзя придумать новый proactive-триггер в
> коде «по ходу».

**Текущий whitelist (утверждён пользователем 2026-04-25):**

| № | Повод | Статус |
|---|---|---|
| — | Reminder fires — явно созданное юзером напоминание | ✅ production |
| Aha-1 | Утреннее напоминание об упомянутом регулярном событии | ✅ via reminder |
| Aha-2 | Предложение меню по ранее упомянутой диете | ✅ production |
| Aha-3 | Предложение добавить часто упоминаемый продукт в список | ⚠️ не реализовано |
| Aha-4 | Recall обещания-факта (≥72ч) | ⚠️ не реализовано |
| Aha-5 | ❌ Вечерний «как прошёл день» | 🗑️ удалён (creepy) |
| Aha-6 | Уточнение после voice-сообщения | ✅ via LLM prompt |

**Запрещено без отдельного утверждения:**
- «Как прошёл день?»
- «Давно тебя не было, всё ли в порядке?»
- «Проверяю, ты занята?»
- Любые ощущения слежки / мониторинга.

**Процесс для нового aha:**
1. Автор описывает повод как Aha-N (в этом документе, draft).
2. Пользователь (Борис) утверждает — запись переходит из draft в whitelist.
3. Только потом реализуется.

## 1. Aha-1 — «Первое утреннее напоминание»

**Статус.** ✅ Работает через обычный reminder-mechanism.

**Триггер.** Юзер во время onboarding упомянул регулярное событие
(кружок, работа, визит). LLM создаёт `FamilyReminder` через
`schedule_reminder` tool с RRULE. Дальше — обычный
`housewife_reminder_worker` шлёт в нужное время.

**Timing.** Вечером за 12 часов до события ИЛИ утром за 45 минут —
зависит от того, что юзер попросил.

**Текст (пример при первом срабатывании):**
```
🔔 Завтра у Маши кружок к 9:00. Форма и сменка готовы?
```

**Кнопки.** Из `ui_labels`:
- `Готово ✅` (callback `rem_done:<id>`)
- `Отложить ⏰` или `Напомни утром ⏰` (callback `rem_snooze:<id>`)

**Код.**
- Создаётся: `services/housewife_chat_tools.py::schedule_reminder`
- Шлётся: `workers/housewife_reminder_worker.py::HousewifeReminderWorker`
- Escalation: после первого fire настраивается re-ping (ограниченно —
  см. `services/housewife_reminders.py::ESCALATION_*`)

**Замечания.**
- LLM-prompt (`_HOUSEWIFE_FOOD_PROMPT`) инструктирует: «если юзер
  описывает повторяющийся паттерн, обязательно через
  `schedule_reminder` с recurrence_rule, а не пачкой one-shots».
- Past-due one-shots (более чем на 15 мин просроченные в момент
  создания) бот не создаёт — отдаёт `skipped:past` (2026-04-23 fix).

## 2. Aha-2 — «Запомнила диету → предложила меню»

**Статус.** ✅ Production (worker + текст).

**Триггер.**
- Юзер прошёл approval (есть `Tenant.approved_at`).
- Прошло **20-48 часов** с approval.
- У юзера в `FamilyMember.notes` есть упоминание диеты/аллергии
  (keyword-match: «диет», «аллерг», «без», «безлакт», «безмолоч»,
  «безглютен», «лактоз», «глютен», «сахарн», «веган», «вегетар»,
  «постн», «не ест», «нельзя»).
- Sentinel-`FamilyReminder` с `source_memo='aha2:<tenant_id>'` не
  существует (idempotency).

**Timing.** 10:00–13:00 UTC (= 13:00–16:00 MSK) — дневное окно,
уже встали, ещё не ужин.

**Текст:**
```
🍽️ У {member_name} есть диета — могу собрать меню на неделю под неё.
Подойдёт всем в семье.
```

**Кнопки** (динамические, через `reply_with_buttons`):
- `Да, собери меню`
- `Не сейчас`
- `Покажи список блюд`

**Код.**
- Worker: `src/sreda/workers/onboarding_aha_worker.py::OnboardingAhaWorker`
- Интеграция: `src/sreda/workers/job_runner.py` (тикает вместе с
  остальными воркерами)
- Sentinel: `FamilyReminder` со `status='fired'`,
  `source_memo='aha2:<tenant_id>'` — создаётся после отправки
  (и если диет нет — тоже создаётся пустой, чтобы не перепроверять
  каждый тик).
- Доставка: через обычный OutboxMessage + `OutboxDeliveryWorker`.

**Замечания.**
- Удалено «Помню, у X есть диета» → просто «У X есть диета» (не
  выпячиваем факт «запоминания»).
- Удалено «Хочешь попробовать?» — кнопки говорят сами.
- Окно 20-48ч: после 48ч уже не первое впечатление, aha теряет смысл.

## 3. Aha-3 — «Частый продукт → предложить в список»

**Статус.** ⚠️ Draft — не реализовано.

**Триггер (draft).** Юзер упомянул один и тот же продукт 2+ раза за
первые 2 дня использования. Внутренний counter бот не раскрывает
юзеру (Фильтр 4 — никакого счёта вслух).

**Timing.** 3-й день утром (10:00 MSK).

**Текст (draft):**
```
🛒 Похоже, молоко у вас часто заканчивается. Добавлю в список
покупок, чтобы не забыть в следующий раз?
```

**Кнопки (draft):**
- `Добавь в список`
- `Не надо`
- `Напомни завтра`

**Требуется реализовать.**
- Детектор повтора упоминаний продукта в `FamilyContext.mentioned_products`.
- Worker (аналогичный `onboarding_aha_worker`).
- Sentinel-запись для идемпотентности.

**Замечания.**
- По фидбеку писателя v1: убрано «ты дважды упоминала молоко» —
  звучит как счёт. Заменено мягким «похоже, часто заканчивается».
- Никаких магазинов / брендов / цен в тексте до интеграции с
  реальным Купер/Я.Лавка (2026-04 — нет API).

## 4. Aha-4 — «Recall обещания через 3 дня»

**Статус.** ⚠️ Draft — не реализовано (высокий cost: нужна NLU).

**Триггер (draft).** 3 дня с момента approval. Юзер упомянул
факт-обещание в первые дни (например, «Надо Машу к стоматологу»),
и этот факт ещё не разрешён (нет связанного reminder/task).

**Timing.** Утро 4-го дня (9:00 MSK).

**Текст (draft):**
```
🧠 Помнишь, в понедельник был разговор про стоматолога для Маши.
Записать напоминание на эту неделю?
```

**Кнопки (draft, динамические):**
- `Да, на пятницу` (ближайший рабочий день)
- `Напишу позже`
- `Уже готово`

**Требуется реализовать.**
- **Новый tool** `flag_unfulfilled_mention(fact, mentioned_at)` —
  LLM во время обычных разговоров помечает факты-обещания.
- **Новая таблица** `unfulfilled_mentions(id, tenant_id, user_id,
  fact_text, mentioned_at, resolved_at)`.
- Worker — смотрит unresolved факты старше 72ч, шлёт recall.
- Обработчик кнопок — resolved_at ставится в now(), создаётся
  reminder или просто закрывается.

**Замечания.**
- Вместо `«ты говорила»` — `«был разговор про X»` (gender-neutral).
- Кнопки динамические — «на пятницу» подставляется ближайший
  рабочий день по факту.

## 5. Aha-5 — ❌ удалён

**Прежний триггер:** 20:00-21:00 первого дня после approval,
«спросила как прошёл день».

**Почему удалён (2026-04-25).** Пользователь: *«Выглядит как будто
агент следит за пользователем. Может вызвать негативное впечатление.
Такого надо избегать.»*

**Закрепление.** Добавлено в Фильтр 5 (анти-сталкер) как постоянное
запрещённое поведение. Не восстанавливать.

## 6. Aha-6 — «Уточнение после voice-сообщения»

**Статус.** ✅ Работает через LLM prompt + `reply_with_buttons`.

**Триггер.** Юзер отправил voice с неоднозначным/неполным
содержимым (например, «завтра к врачу в 11» — непонятно, кто и
к какому врачу).

**Timing.** Немедленно, в ответ на voice.

**Текст (генерируется LLM по prompt-правилу):**
```
📝 Услышала: «Завтра к врачу в 11»
Кто идёт и к какому врачу? Я поставлю напоминание утром за 30 мин.
```

**Кнопки (динамические, из `FamilyContext`):**
- `Пете к педиатру` (если Петя есть в семье и у него упоминался педиатр)
- `Маше к ортодонту` (если Маша + ортодонт)
- `Другое — напишу`

**Код.**
- STT: `services/speech/*` (Groq + Yandex fallback).
- LLM-turn: `execute_conversation_chat` в `runtime/handlers.py`.
- Кнопки: tool `reply_with_buttons` + callback `btn_reply:<token>`.
- Prompt-правило: блок «Правила кнопок» в `_HOUSEWIFE_FOOD_PROMPT`:

> Если вопрос подразумевает имя/выбор из списка — делай кнопки
> персонализированными («Пете — к педиатру», «Маше — к ортодонту»),
> а не абстрактными («Ребёнок 1», «Ребёнок 2»).

**Замечания.**
- Эталон gender-neutral — кнопки используют имена из памяти, не
  «ты имел(-а) в виду X».
- Aha-6 формально не «proactive» (реакция на юзера), но
  включён в whitelist как особый случай умного read-back.

## 7. Общие правила для новых aha-moments

При добавлении нового повода (Aha-N) — обязательно:

### Перед кодом
- [ ] Описать в этом файле: триггер / timing / текст / кнопки
- [ ] Получить явное утверждение от пользователя (добавление в whitelist §0)
- [ ] Проверить что триггер НЕ про «мониторинг активности» юзера
- [ ] Проверить что текст не содержит запрещённых фраз из Фильтра 5
- [ ] Обдумать идемпотентность (как не слать дважды)

### В коде
- [ ] Новый worker (либо ветка в существующем `onboarding_aha_worker`)
- [ ] Sentinel-запись для идемпотентности (паттерн `source_memo='ahaN:<id>'`)
- [ ] Окно времени отправки (чтобы не будить ночью)
- [ ] Кнопки через `reply_with_buttons` или хардкод-callback
- [ ] Prompt-инструкция (если LLM должна что-то настраивать под aha)

### После деплоя
- [ ] `test_anti_stalker_tone.py` green (автоматически по файлу воркера)
- [ ] Запись в этом документе переведена из ⚠️ draft в ✅ production
- [ ] Записать первое прод-срабатывание в логе — что увидел юзер

## 8. Фильтры тона (обязательны для ВСЕХ aha)

Применяются к каждому тексту сверху:

1. **Фильтр 1 — прозрачность «зачем».** Если aha запрашивает данные
   — объяснить почему (см. Aha-2 «Подойдёт всем в семье»).
2. **Фильтр 2 — progressive disclosure.** Не просить всё сразу.
3. **Фильтр 3 — лёгкий отказ.** Всегда кнопка «Не сейчас» / «Позже».
4. **Фильтр 4 — read-back без счёта.** Никаких «ты N раз», «за
   последние N дней».
5. **Фильтр 5 — proactive только по whitelist.** Если не в §0 — не
   реализуем.
6. **Фильтр 6 — opt-in на память.** При первом сохранении факта
   бот говорит «записала X, скажи "забудь про X" чтобы убрать».
7. **Gender-neutral.** Никакого ж.р. в прошедшем времени про
   юзера. Про Среду — ж.р. разрешён.
